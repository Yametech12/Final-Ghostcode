-- Google Cloud Storage Setup for User Uploads
-- This is a guide for setting up GCS bucket equivalent to Supabase storage

-- Prerequisites:
-- 1. GCP Project with Cloud Storage API enabled
-- 2. Service Account with Storage Admin or Object Admin role
-- 3. For Vercel: Workload Identity Federation configured
-- 4. Environment variables set in Vercel or .env:
--    GCP_PROJECT_ID, GCP_BUCKET_NAME, GCP_PROJECT_NUMBER,
--    GCP_SERVICE_ACCOUNT_EMAIL, GCP_WORKLOAD_IDENTITY_POOL_ID,
--    GCP_WORKLOAD_IDENTITY_PROVIDER_ID
--    For local: GOOGLE_APPLICATION_CREDENTIALS=path/to/key.json

-- Step 1: Create the bucket
-- Run: npx tsx api/create-bucket-gcs.ts
-- Or manually in GCP Console: Storage > Buckets > Create bucket
-- Bucket name: user-uploads (or as per GCP_BUCKET_NAME)
-- Location: Choose appropriate region
-- Storage class: Standard
-- Access control: Uniform (recommended for fine-grained control)

-- Step 2: Configure CORS (if using bucket.create in script, it's done)
-- In GCP Console: Bucket > Configuration > CORS configuration
-- Add:
[
  {
    "origin": ["*"],
    "method": ["GET", "POST", "PUT", "DELETE"],
    "responseHeader": ["Content-Type"],
    "maxAgeSeconds": 3600
  }
]

-- Step 3: Make bucket public for read access
-- In GCP Console: Bucket > Permissions > Add
-- Grant Storage Object Viewer to allUsers

-- Step 4: IAM Policies for uploads
-- The service account needs Storage Object Admin on the bucket
-- In GCP Console: Bucket > Permissions > Add
-- Add the service account email with Storage Object Admin role

-- Step 5: For user-specific uploads
-- Use signed URLs generated by api/upload.ts
-- Frontend requests upload URL with userId, then uploads directly to GCS
-- Files are stored in userId/filename structure for isolation

-- Note: Unlike Supabase RLS, GCS uses signed URLs for secure uploads
-- No direct equivalent to RLS, but signed URLs provide similar security
